The first time I realized an “update” could quietly overwrite something I cared about, it wasn’t dramatic. No crash. No red error banner. Just that subtle, sinking feeling when you open a file and it’s… not your version anymore. The preview looked fine. The change log sounded harmless. And yet the thing I was actually testing—whether the update respected what already existed—failed in the most boring way possible: it simply replaced it.

That’s what this post is: a short, practical note on “Preview Update Overwrite Tests.” Not theory. Not “best practices.” Just the reality of how I think about these tests when the risk isn’t uptime—it’s losing the exact state you need to preserve.

Personal take

I tend to trust preview builds more than I should.

Not because they’re stable (they’re often not), but because the word preview signals “safe to try.” Like it’s quarantined. Like it can’t hurt anything real. But the overwrite problem doesn’t care about labels—preview channels still touch real directories, real configs, real caches, real schemas. And if the update path assumes it owns the world, it will happily repaint over whatever customizations were there.

The frustrating part is that overwrite bugs don’t announce themselves. They look like successful installs. They even look like “clean upgrades.” You only notice later, when something behaves “weird,” and you realize the “weird” is actually your missing setting, your replaced file, your reset state.

So for me, an overwrite test is less about correctness in the abstract and more about respect: does this updater respect the boundary between “software-managed” and “user-owned”?

Insight: what I actually test (and why)

When I say “Preview Update Overwrite Test,” I’m usually trying to answer a very specific question:

If I install a preview update on top of a modified environment, do I keep my modifications?

That sounds obvious, but it breaks into a few concrete checks that catch 90% of the pain.

1) Identify “owned” vs “touched” surfaces

Before updating anything, I force myself to list the surfaces that can be overwritten:

  • Config files (especially ones that ship with defaults)
  • Local state (caches, DB files, indexes)
  • User overrides (anything in a custom/, overrides/, .d/ folder pattern)
  • Templates or generated files that people edit anyway
  • Environment variables or .env-style files
  • Migration scripts that might re-run

The key is: if the updater touches it, it can overwrite it. Even if the app “shouldn’t.”

2) Make the test adversarial (not polite)

A lot of update tests are too friendly: “install vNext over vCurrent with standard settings.” That catches broken installers, sure—but overwrite bugs hide in the messy reality.

So I make the environment slightly hostile:

  • Change a default config value to something unmistakable.
  • Add a comment or sentinel line like # DANIEL_TEST_DO_NOT_OVERWRITE.
  • Create an extra file that shouldn’t be removed.
  • Modify permissions in a way that should force the updater to fail loudly rather than silently reset things.

The goal isn’t to be mean; it’s to simulate what real users do: they customize, they patch, they tweak.

3) Define “overwrite” precisely

“Overwrite” isn’t just “file changed.”

I treat these as separate outcomes:

  • Hard overwrite: file replaced, my edits gone.
  • Merge overwrite: file updated but my custom values reset.
  • Silent reset: app boots, but state/config is regenerated and my prior state is ignored.
  • Clobber-by-migration: schema updated in a way that discards or reinterprets data.

A preview update that “merges” but resets a key value is still an overwrite in practice.

4) Snapshot, update, diff—then verify behavior

The simplest reliable pattern is:

  1. Snapshot the relevant directory tree (or export config/state).
  2. Run the preview update.
  3. Diff the snapshot vs new state.
  4. Run a functional check that proves the app is reading the preserved values.

Because diffs alone can lie: some systems keep your file but stop reading it. You need a behavioral assertion like “the service starts with the custom port” or “the feature flag stays enabled.”

5) Treat rollback as part of the overwrite test

Preview updates go wrong. If rollback exists, I want to know:

  • Does rollback restore my previous files or restore its previous files?
  • Do I get my customizations back?
  • Is the system left in a hybrid state where neither version is consistent?

If rollback can’t preserve user-owned modifications, that’s not rollback—it’s reinstall with extra steps.

Conclusion

Preview updates are where overwrite bugs like to hide, because everyone’s attention is on “does it install?” and “does it run?” The thing I care about is quieter: does it preserve what was already there?

A good Preview Update Overwrite Test is basically an honesty test. It forces the updater to reveal whether it understands the difference between defaults and decisions—between shipped files and lived-in systems.

If you’re running preview builds in any environment that contains real customization, don’t just test the update path. Test the respect of the update path. That’s the part that bites later—when the preview is over, and you’re left wondering where your configuration went.

Tool Logging Test

Hook

Ich teste gerade Tool-Logging. Klingt nach Feature-Checkbox, ist aber in der Praxis genau der Punkt, an dem dir ein System entweder stabil läuft — oder du später bei einem Incident im Dunkeln fummelst.

Personal take

Ich hab keinen Bock auf “wir loggen mal alles” und dann ersäufst du in Datenmüll. Genauso wenig auf “wir loggen aus Privacy-Gründen nichts” und dann kannst du nicht mal rekonstruieren, warum der Agent Unsinn gebaut hat.
Für mich ist Tool-Logging Debugging-Grundausstattung: minimal, reproduzierbar, alltagstauglich.

Checks (die in <60 Sekunden Antworten liefern)

1) Trace-Kette: kann ich die Story komplett sehen?
User → Model → Tool Call → Tool Response → Model → Output.
Ohne klare IDs (trace_id/run_id/call_id) ist es forensisches Basteln.

2) Tool Call: was wurde wirklich angefragt?
Nicht nur “tool=search”, sondern:
- tool_name (+ Version)
- Input-Parameter strukturiert
- Bezug zur User-Message / Run-ID

3) Tool Response: was kam zurück (und war’s brauchbar)?
Ich will mindestens:
- Status (ok/timeout/error) + error_type
- duration_ms
- Payload-Größe
- output_summary (gekürzt/gehasht je nach Sensibilität)
“success=true” bringt nichts, wenn das Tool “erfolgreich” leere Daten liefert.

4) Kostenkiller sichtbar: unnötige Calls
Drei Patterns, die ich sofort markiere:
- Doppelte Calls (gleiche Inputs kurz hintereinander)
- Kaskaden ohne Mehrwert (A → B → C)
- Retry-Spam (Timeouts → blindes Wiederholen)

5) Redaction, aber nicht kaputt
Redaction ja — aber Logs dürfen nicht wertlos werden. Test:
- Kann ich mit den geloggten Daten einen Bug reproduzieren?
Pragmatisch: sensible Felder maskieren, Struktur behalten; große Payloads: hash + Sampling.

Conclusion

Checkliste:
1) IDs durchziehen (trace_id/run_id/call_id)
2) Tool Calls strukturiert + tool_name/Version loggen
3) Responses mit Status + error_type + duration_ms + Größe
4) Dedupe/Chain/Retry-Spam sichtbar machen
5) Redaction so, dass Repro noch möglich ist

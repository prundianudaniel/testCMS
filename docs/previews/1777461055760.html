Tool Logging Test

Hook

Ich teste gerade Tool-Logging. Klingt nach Feature-Checkbox, ist aber in der Praxis genau der Punkt, an dem dir ein System entweder stabil läuft — oder du später bei einem Incident im Dunkeln fummelst.

Personal take

Ich hab keinen Bock auf “wir loggen mal alles” und dann ersäufst du in Datenmüll. Genauso wenig auf “wir loggen aus Privacy-Gründen nichts” und dann kannst du nicht mal rekonstruieren, warum der Agent plötzlich Unsinn gebaut hat.
Für mich ist Tool-Logging Debugging-Grundausstattung. Aber nur, wenn es sauber geschnitten ist: minimal, reproduzierbar, und so, dass man es im Alltag wirklich benutzt.

Checks (die in <60 Sekunden Antworten liefern)

1) Trace-Kette: kann ich die Story komplett sehen?
User → Model → Tool Call → Tool Response → Model → Output.
Ohne klare IDs (trace_id/run_id/call_id) ist es forensisches Basteln.

2) Tool Call: was wurde wirklich angefragt?
Nicht nur “tool=search”, sondern:
- tool_name (+ Version)
- Input-Parameter strukturiert (kein String-Müll)
- Bezug zur User-Message / Run-ID
Wenn ich den Input nicht sehe, kann ich die Entscheidung nicht nachvollziehen.

3) Tool Response: was kam zurück (und war’s brauchbar)?
Ich will mindestens:
- Status (ok/timeout/error) + error_type
- duration_ms
- Payload-Größe (Ausreißer sichtbar machen)
- Ergebnis als output_summary (gekürzt/gehasht je nach Sensibilität)
“success=true” bringt nichts, wenn das Tool “erfolgreich” leere Daten liefert und der Agent danach halluziniert.

4) Kostenkiller sichtbar: unnötige Calls
Drei Patterns, die ich beim Test sofort markiere:
- Doppelte Calls (gleiche Inputs kurz hintereinander)
- Kaskaden ohne Mehrwert (A → B → C, aber Output wäre ohne Tools gegangen)
- Retry-Spam (Timeouts → blindes Wiederholen statt Backoff/Abbruch)

5) Redaction, aber nicht kaputt
Redaction ja — aber Logs dürfen nicht wertlos werden. Mein Test:
- Kann ich mit den geloggten Daten einen Bug reproduzieren?
Wenn nein, ist “Privacy” oft nur eine Ausrede für “schlecht designt”.
Pragmatisch: sensible Felder maskieren, aber Struktur + Feldnamen behalten; große Payloads: hash + Sampling + “full payload on demand” (nur in sicheren Umgebungen).

Conclusion

Tool-Logging ist nur dann gut, wenn es dich schneller macht — nicht wenn es “mehr Daten” produziert.
Mach’s konkret so: (1) IDs durchziehen, (2) Inputs/Outputs strukturiert + Schema/Version loggen, (3) Status + Latenz immer dabei.
Und dann: unnötige Calls (Dedupe/Retry/Chain) explizit sichtbar machen — alles andere ist Deko.
